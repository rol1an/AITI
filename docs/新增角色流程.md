# ACGTI 新增角色流程

本文档用于说明在当前仓库中新增一个角色时，需要改哪些文件、按什么顺序改、改完后如何自查。

## 1. 先理解角色在项目里的落点

当前项目里，一个"角色"的数据源是**一个独立 JSON 文件**：

- `src/content/characters/<id>.json`
  角色的完整配置，包含核心资料、视觉配置和多语言翻译。

构建时，脚本会从所有 `<id>.json` 自动生成前端使用的聚合文件：

- `src/data/characters.json` — 角色核心资料数组（由脚本生成）
- `src/data/characterVisuals.json` — 角色视觉配置映射（由脚本生成）

此外，角色还涉及以下**手动维护**的文件：

- `public/images/characters/`
  角色立绘图片资源。运行时优先使用 `WebP`，仓库中也允许保留透明 `PNG` 源图；角色必须有图，且应为透明背景素材。
- `public/images/characters/thumbs/`
  角色缩略图，**同样为 WebP 格式**，宽度 400px，用于图鉴页卡片展示。
- `src/data/characterProbabilities.json`
  角色命中概率配置，用于结果页展示"人群命中率"。
- `src/i18n/characters.ts`
  角色名和作品名的多语言映射表（由构建脚本读取校验，需手动维护）。

运行时的关系如下：

1. `quizEngine.ts` 根据答题结果对 `characters.json` 中的角色进行排序。
2. `characterVisuals.ts` 用 `characterVisuals.json` 中的图片（全尺寸 + 缩略图）和主题色补全角色展示信息。
3. `characterProbability.ts` 从 `characterProbabilities.json` 读取该角色的命中率。
4. `src/i18n/characters.ts` 为角色名和作品名提供 `zh-CN`、`zh-TW`、`en`、`ja` 的展示文本。

所以，新增角色需要保证"源文件、素材（全尺寸+缩略图）、概率、i18n"四处能对上同一个 `id`。

同时，本项目对新增角色还有以下硬性要求：

- 角色必须配套图片，且图片必须处理成透明图层。运行时默认优先使用 `WebP`，如素材处理阶段需要保留 `PNG` 也可以共存。
- 角色必须同时提供**全尺寸图**和**缩略图**（400px 宽），分别用于结果页和图鉴页。
- 每次新增角色后，都必须基于最新角色库全局重跑一次稀有度（命中概率）数据，不能只手动补一个值。
- 新增角色后运行 `npm run validate:data` 确保数据完整。

## 2. 角色源文件结构

每个角色在 `src/content/characters/` 下有一个独立 JSON 文件，文件名即角色 `id`，例如 `src/content/characters/hatsune-miku.json`。

结构如下：

```json
{
  "meta": {
    "id": "hatsune-miku",
    "name": "初音未来",
    "series": "VOCALOID",
    "matchCode": "ENFP",
    "matchCodeFlex": ["ENFJ", "ESFJ"],
    "code": "HTMK",
    "title": "世界第一的公主殿下（劳务派遣版）",
    "archetypeId": "luminous-lead",
    "tags": ["电子韭菜收割机", "葱田永动机", "演唱会空气吉他冠军", "二次元公务员"],
    "note": "结果页角色解析文案。",
    "vector": {
      "expression": 0.59,
      "temperature": 0.39,
      "judgement": 0,
      "order": 0,
      "agency": 0.59,
      "aura": 0.39
    },
    "matchWeight": 1.0599
  },
  "visual": {
    "image": "/images/characters/初音未来.webp",
    "thumb": "/images/characters/thumbs/初音未来.webp",
    "accent": "#39c5bb"
  },
  "i18n": {
    "name": {
      "zh-CN": "初音未来",
      "zh-TW": "初音未來",
      "en": "Hatsune Miku",
      "ja": "初音ミク"
    }
  }
}
```

各字段说明：

### meta（核心资料）

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 全仓库唯一标识，英文小写加连字符，例如 `kaname-madoka` |
| `name` | 是 | 角色展示名 |
| `series` | 是 | 作品名 |
| `matchCode` | 是 | 底层 MBTI 映射，例如 `INFP` |
| `code` | 是 | 对外展示的角色自定义代码，结果页核心标签 |
| `title` | 否 | 结果页称号 |
| `archetypeId` | 是 | 原型 ID，必须从以下 8 个中选 |
| `tags` | 是 | 标签数组，结果页展示 |
| `note` | 是 | 角色解析文案 |
| `vector` | 是 | 六维向量，六个轴必须写全 |
| `matchCodeFlex` | 否 | 弹性兼容 MBTI 列表 |
| `matchWeight` | 否 | 概率权重因子 |
| `personaBasis` | 否 | 数据来源说明 |
| `signature` | 否 | 独特判别点和题目亲和度 |

`archetypeId` 可选值：

- `luminous-lead`
- `icebound-observer`
- `oathbound-captain`
- `trickster-orbit`
- `gentle-healer`
- `shadow-strategist`
- `chaos-spark`
- `moonlit-guardian`

`vector` 六个轴：

- `expression`：表达外放程度
- `temperature`：情感温度
- `judgement`：判断/决断倾向
- `order`：秩序与规划倾向
- `agency`：主动性与推动力
- `aura`：存在感与气场

### visual（视觉配置）

| 字段 | 说明 |
|------|------|
| `image` | 全尺寸图路径，用于结果页主视觉和海报导出 |
| `thumb` | 缩略图路径，用于角色图鉴页卡片展示 |
| `accent` | 角色主题色，结果页顶部色块、按钮和强调区会用到 |

### i18n（多语言翻译）

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 强烈建议 | 角色名四语翻译，缺失时回退到 `meta.name` |
| `series` | 条件必填 | 仅当角色来自新作品时需要补（已收录的作品可省略） |

每个翻译对象需提供四个 locale：`zh-CN`、`zh-TW`、`en`、`ja`。

## 3. 新增角色的标准流程

推荐按下面顺序操作，避免中途出现引用断裂。

### 步骤 1：确定角色基础方案

在落文件前，先把下面这些内容定下来：

- `id`：全仓库唯一，英文小写加连字符
- `name`：角色展示名
- `series`：作品名
- `code`：对外展示的角色代码，确保不与已有角色重复
- `matchCode`：底层 MBTI
- `archetypeId`：原型
- `vector`：六维向量
- `tags`、`title`、`note`：结果页展示文案

### 步骤 2：创建角色源文件

在 `src/content/characters/` 下新建 `<id>.json`，填入完整数据。

> 参考现有角色文件（如 `hatsune-miku.json`）的结构，不要随意发明新字段。

注意事项：

- `id` 必须唯一，并且文件名与 `id` 一致。
- `matchCode` 必须符合 MBTI 四字母格式。
- `vector` 六个轴必须写全，值在 `[-1, 1]` 范围内。
- `code` 不要和已有角色重复。
- `i18n.name` 强烈建议同时补全四个语言的翻译。

可选增强字段：

- `matchCodeFlex`：允许角色兼容多个 MBTI。
- `signature.uniqueAxes`：强调该角色在六维向量上的独特判别点。
- `signature.questionAffinity`：把某些题目答案倾向额外绑定到该角色，提高辨识度。

`questionAffinity` 格式示例：

```json
{
  "signature": {
    "questionAffinity": [
      { "questionId": "q12", "expected": "agree", "weight": 1.4 },
      { "questionId": "q20", "expected": "neutral", "weight": 1.2 }
    ]
  }
}
```

### 步骤 3：放入角色图片资源（PNG → WebP）

#### 3a. 准备原始素材

把角色原始图片（PNG 格式，透明背景）放到：

`public/images/characters/`

建议遵循当前仓库现状：

- 文件名尽量与角色中文名一致，便于人工管理。
- 必须使用透明背景的人物立绘。
- 构图尽量适配结果页头部大图展示，避免头部被裁切过多。

如果原图不是透明背景，新增角色前应先完成抠图处理。仓库中已有相关脚本：

- `scripts/remove_character_backgrounds.py`
- `scripts/process_character_assets.py`

#### 3b. 转换为 WebP 并生成缩略图

使用项目提供的转换脚本，一键生成全尺寸 WebP 和缩略图：

```bash
python scripts/generate_character_webp.py
```

该脚本会：

- 将 `public/images/characters/*.png` 转为同目录下的 `*.webp`（全尺寸，quality 85）
- 同时在 `public/images/characters/thumbs/` 下生成 400px 宽的缩略图（quality 80）

也可以手动用 ffmpeg 转换：

```bash
# 全尺寸 WebP
ffmpeg -i "角色名.png" -quality 85 "角色名.webp"

# 缩略图（400px 宽）
ffmpeg -i "角色名.png" -vf "scale=400:-1" -quality 80 "thumbs/角色名.webp"
```

注意事项：

- 最终部署的是 WebP 格式，**不是 PNG**。
- 全尺寸图和缩略图缺一不可。
- 缩略图固定 400px 宽，高度按比例缩放。
- 保留 PNG 源图以备后续重新处理，但运行时不使用。

### 步骤 4：补充角色命中概率

新增角色后，不应只在 `src/data/characterProbabilities.json` 里手填一个值，而是要基于最新全量角色库重新生成整份稀有度数据。

当前仓库里已有模拟脚本：

- `scripts/probability-simulation.mjs`

标准要求：

- 每次新增角色，都必须全局重跑一次稀有度。
- 重跑后，整体回写 `src/data/characterProbabilities.json`。
- 不建议只追加一条 `"new-character-id": x`，因为这会让整体分布失真。

最终 `src/data/characterProbabilities.json` 中的 `probabilities` 会是类似下面的结果：

```json
"new-character-id": 1.25
```

这里的值用于结果页展示"该角色在人群中的命中比例"，运行时由 `src/utils/characterProbability.ts` 读取。

注意事项：

- 所有角色最好都有值，否则前端会回退为 `0`。
- 这是展示用数据，不会直接参与角色排序，但会影响结果页可信度和完整性。
- 稀有度必须以"全量角色一起重算"为准，不能把新增角色当成局部补丁处理。

### 步骤 5：补齐 i18n 映射（手动维护）

虽然角色源文件中已有 `i18n.name`，但 `src/i18n/characters.ts` 仍需手动维护（它是前端运行时直接引用的 TypeScript 文件，无法由脚本自动生成）。

编辑 `src/i18n/characters.ts`，至少补两类映射：

- `characterNameI18n`
  为新角色的 `id` 增加多语言名称。
- `seriesI18n`
  如果新角色来自一个之前未收录的新作品，也要为该作品补多语言名称。

角色名示例：

```ts
'new-character-id': {
  'zh-CN': '角色简中名',
  'zh-TW': '角色繁中名',
  en: 'Character English Name',
  ja: 'キャラクター名',
}
```

作品名示例：

```ts
'作品名': {
  'zh-CN': '作品名',
  'zh-TW': '作品繁中名',
  en: 'Series English Name',
  ja: '作品日本語名',
}
```

注意事项：

- 这一步不是可选项，新增角色默认需要做 i18n 适配。
- 如果只在源文件里写中文名，而不补 `src/i18n/characters.ts`，英文、日文、繁中环境下会退回原始值，展示会不完整。
- 若角色所属作品是已有作品，只需复用已有 `seriesI18n` 项；若是新作品，必须新增该作品的多语言映射。
- `characterNameI18n` 的 key 必须与角色 `id` 完全一致。

### 步骤 6：构建并校验

运行校验和构建：

```bash
npm run validate:data
```

校验脚本会检查：

- `id` / `code` 唯一性
- `matchCode` 是否合法 MBTI
- `vector` 六维是否齐全
- `visual` 是否存在
- `i18n.name` 是否缺失
- 必填字段完整性

如果校验通过（0 错误），继续运行完整构建：

```bash
npm run build
```

构建会自动：

1. 校验角色数据
2. 从 `src/content/characters/*.json` 生成聚合的 `characters.json` 和 `characterVisuals.json`
3. 运行 TypeScript 编译和 Vite 构建

## 4. 新增角色时的判定建议

为了让新增角色在结果上"像这个角色"，而不是只挂一个名字，建议按下面逻辑补数据。

### 4.1 `matchCode` 怎么定

- 先定底层 MBTI。
- 这个值决定角色在大方向上的命中范围。
- 如果角色存在边界模糊区，可以加 `matchCodeFlex` 做弹性兜底。

### 4.2 `archetypeId` 怎么定

- `matchCode` 决定底层人格型。
- `archetypeId` 决定角色更接近哪一种二次元原型表现。
- 两者最好保持一致，不要出现明显互相冲突的设定。

可以直接参考 `src/utils/quizEngine.ts` 中 `TYPE_TO_ARCHETYPE` 的映射关系。

### 4.3 `vector` 怎么定

建议做法：

- 先找 1 到 2 个仓库里最接近的已有角色作参照。
- 不要一上来把所有值都拉满。
- 真正有辨识度的地方，再通过 `signature.uniqueAxes` 或 `questionAffinity` 强化。

### 4.4 `note` 和 `tags` 怎么写

虽然项目是二次元包装，但结果页文案不能只剩梗图。

建议保持这个平衡：

- `title` 负责记忆点和角色称号。
- `tags` 负责快速识别特征。
- `note` 负责把"人格倾向 + 角色观感"串起来。

不要把角色描述写成医学、诊断或心理治疗口吻。

## 5. 改完后的自查清单

至少检查下面这些点：

- `src/content/characters/<id>.json` 结构合法，JSON 无语法错误。
- `id` 在所有文件中完全一致（文件名 = meta.id）。
- `visual` 包含 `image`、`thumb`、`accent` 三个字段。
- `public/images/characters/` 中存在对应的全尺寸 WebP 图片。
- `public/images/characters/thumbs/` 中存在对应的缩略图 WebP。
- 图片已经完成抠图，背景为透明图层。
- `src/data/characterProbabilities.json` 已补该角色概率。
- `src/data/characterProbabilities.json` 是基于全量角色重新生成后的结果，而不是手工局部追加。
- `src/i18n/characters.ts` 已补角色名的多语言映射。
- 如果角色来自新作品，`src/i18n/characters.ts` 也已补作品名的多语言映射。
- `matchCode` 为合法 MBTI。
- `archetypeId` 为合法原型 ID。
- `vector` 六轴齐全，值在 `[-1, 1]` 范围内。
- `code` 不与已有角色重复。
- `questionAffinity` 引用的题号真实存在。
- `npm run validate:data` 通过（0 错误）。

## 6. 建议的验证流程

新增角色后，建议至少做一次最小验证：

```bash
npm run validate:data
npm run build
```

然后人工检查：

1. 启动本地站点，进入角色图鉴页，确认缩略图能正常显示，没有 404。
2. 点击该角色进入结果页，确认全尺寸图能正常显示。
3. 确认图片已经是透明图层，没有白底、实底或明显抠图毛边。
4. 确认结果页主色、标题、标签、解析文案显示正常。
5. 切换至少一种非简中语言，确认角色名和作品名没有漏翻或退回异常值。
6. 如有调试入口或定向测试方式，检查该角色是否能被正常命中。

如果后续要做更稳的内容扩充，建议再补两类验证：

- 用固定 MBTI 样本检查该角色是否落在预期区间。
- 检查全局重跑后的稀有度分布是否合理，避免角色库扩充后命中率失真。

## 7. 一次典型新增的实际改动清单

一次完整的新增角色，通常会改这几处：

- `src/content/characters/<id>.json`（新增角色源文件）
- `src/i18n/characters.ts`（补角色名和作品名翻译）
- `public/images/characters/角色名.webp`（全尺寸 WebP）
- `public/images/characters/thumbs/角色名.webp`（缩略图 WebP）
- `src/data/characterProbabilities.json`（全局重算后更新）

**以下文件由构建脚本自动生成，不要手动编辑：**

- `src/data/characters.json`
- `src/data/characterVisuals.json`

如果还涉及素材处理或概率重算，可能还会临时使用：

- `scripts/process_character_assets.py`（抠图预处理）
- `scripts/remove_character_backgrounds.py`（去背景）
- `scripts/generate_character_webp.py`（PNG → WebP + 缩略图）
- `scripts/probability-simulation.mjs`（全局重跑稀有度）

## 8. 最后原则

新增角色时，优先保证以下三件事：

- 角色代码对外展示明确。
- 底层 MBTI 与原型映射自洽。
- 文案、视觉、概率、素材四部分保持一致。

此外，下面三条应视为默认强约束：

- 没有透明抠图图片和 WebP 缩略图的角色，不算完整接入。
- 没有在新增后全局重跑稀有度的角色，不算完整接入。
- 没有完成 i18n 适配的角色，不算完整接入。

如果只补了名字和图片，没有补齐向量、原型、概率和结果文案，这个角色在当前项目里就还不算完整接入。
